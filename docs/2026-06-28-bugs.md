## 2. 详细 Bug 列表

### 🔴 B1 — ReorderWindow 截止时间计算错误

- **来源**: `code-analysis-modification-plan.md` §二.1.3
- **位置**: `src/domain/services/reorder_window.rs:42`
- **状态**: open
- **违反**: 红线 2.2「乱序场景按策略重排」+ AGENTS.md「window + 挂钟」语义
- **症状**:
  ```rust
  let newest_key = *self.buffer.last_key_value().unwrap().0;
  let cutoff = newest_key.saturating_sub(self.window_ms as i64);
  ```
  截止时间从消息排序键（sequence / timestamp）推导，**而非挂钟时间**。
- **触发场景**:
  - seq=100 后突然 seq=999999 消息到达 → cutoff = 999999 - 200 = 999799
  - 中间 seq=200~999798 的消息被错误提前刷新
- **修复方案**: 记录消息实际插入时间戳 (`Instant::now()`)，使用挂钟时间计算 cutoff
- **工作量**: 4h
- **关联测试**: 需新增 `test_cutoff_based_on_wall_clock`
- **修复位置**: `2026-06-28-optimization-backlog.md` OP-P2-2

### 🔴 B2 — Conversation 聚合边界被破坏（clone 泄漏）

- **来源**: `code-analysis-modification-plan.md` §二.1.2
- **位置**: `src/adapters/conversation_store.rs:59-79`
- **状态**: open
- **违反**: 红线 2.1「Conversation 为一等对象」+ DDD「聚合外部只能通过聚合根方法修改」
- **症状**:
  ```rust
  let ctx = PipelineContext {
      message: m,
      conversation: conversation.clone(),  // BUG: 克隆的聚合
      ...
  };
  ```
- **影响**:
  - 中间件对克隆的任何修改**不会反映回真实聚合**
  - 中间件可绕过聚合根直接读/写 Conversation
- **修复方案**: 创建 `ConversationSnapshot` 只读值对象

  ```rust
  pub struct ConversationSnapshot {
      pub route_key: RouteKey,
      pub conversation_id: String,
      pub peer_id: String,
      pub conversation_type: ConversationType,
      pub message_count: u64,
      pub last_active_at: i64,
  }
  // PipelineContext.conversation → conversation_snapshot
  ```
- **工作量**: 5h
- **修复位置**: `2026-06-28-optimization-backlog.md` OP-P2-1

### 🔴 B3 — 熔断器 HalfOpen 状态永久卡死

- **来源**: `code-analysis-modification-plan.md` §二.1.1
- **位置**: `src/core/resilience/circuit_breaker.rs:73`
- **状态**: open
- **违反**: 红线 2.5「Circuit Breaker 必须有自恢复路径」+ 云原生韧性约束
- **症状**:
  ```rust
  CircuitState::HalfOpen => {
      self.failure_count.load(Ordering::SeqCst) < self.config.half_open_max
  }
  ```
  当 `failure_count >= half_open_max` 时，`allow_request()` 永久返回 `false`，**没有任何超时机制从 HalfOpen 转回 Open**。
- **影响**: 熔断器永久卡死在 HalfOpen，所有依赖它的调用（AI / Send）静默失败
- **修复方案**:
  1. 在 `allow_request()` 的 HalfOpen 分支检查 `opened_at` 超时
  2. 超时后重置 `failure_count` → 转回 Open → 重新等待完整 timeout
- **工作量**: 3h
- **关联测试**: 需新增 `test_halfopen_timeout_transition`
- **修复位置**: `2026-06-28-optimization-backlog.md` OP-P0-2
